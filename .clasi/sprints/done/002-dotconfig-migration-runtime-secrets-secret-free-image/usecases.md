# Sprint 002 Use Cases

## SUC-001: Operator initialises dotconfig and migrates existing secrets

- **Actor**: Operator / Deployer
- **Trigger**: Operator retires git-crypt and switches to SOPS-encrypted dotconfig layout.
- **Preconditions**: `dotconfig` CLI is installed; existing `config/` tree is present and git-crypt-decrypted locally.
- **Main Flow**:
  1. Operator runs `dotconfig init` to generate an age keypair and write `config/sops.yaml`.
  2. Operator migrates values from the current flat layout (`config/config.env`, `config/prod.env`, `config/secrets/secret.env`, `config/secrets/prod.env`, etc.) into `config/{deploy}/public.env` (non-secret) and `config/{deploy}/secrets.env` (SOPS-encrypted).
  3. Operator removes the git-crypt filter from `.gitattributes` and stops tracking raw secret files.
  4. `dotconfig audit` reports zero unencrypted secrets.
- **Postconditions**: All secrets are SOPS-encrypted at rest; no plaintext secret files remain tracked by git.
- **Acceptance Criteria**:
  - [ ] `config/sops.yaml` exists and age keypair is generated.
  - [ ] All deploy environments (devel, prod, local-prod) have corresponding dotconfig layout files.
  - [ ] `.gitattributes` git-crypt rule removed.
  - [ ] `dotconfig audit` exits 0.

## SUC-002: Operator generates a deploy env-file and deploys to prod

- **Actor**: Operator / Deployer
- **Trigger**: Operator runs `make redeploy` to ship a new image and restart the prod stack.
- **Preconditions**: dotconfig is initialised; age secret key is available on the deploy machine; `config/prod/secrets.env` is SOPS-encrypted and present.
- **Main Flow**:
  1. Operator (or Makefile) runs `dotconfig load -d prod --no-export -e -o .env` â€” decrypts and assembles all prod config, base64-embeds `id_rsa` as `ID_RSA=<b64>`, writes `.env` (no `export` prefix).
  2. `make build` â€” Docker build produces a secret-free image (no `COPY config/secrets/*` lines).
  3. `make up` â€” Docker Swarm deploys the stack; the `codeserver` service picks up the env-file via `env_file: .env`.
  4. Container starts; entrypoint shell wrapper decodes `$ID_RSA` to `/root/.ssh/id_rsa` (chmod 600) and unsets the variable.
- **Postconditions**: Running container has all secrets at runtime; Docker image contains no secrets in any layer.
- **Acceptance Criteria**:
  - [ ] `docker history <image>` shows no secret values in any layer.
  - [ ] Container starts and SSH to a worker node succeeds (real host-start).
  - [ ] `make redeploy` runs end-to-end without manual intervention.

## SUC-003: App loads correct configuration in each environment

- **Actor**: App process (automated, triggered by container or local start)
- **Trigger**: Application starts (gunicorn via supervisord, or `flask run` locally).
- **Preconditions**: For prod/local-prod, `.env` is generated by `dotconfig load`. For devel, `.env` is generated by `dotconfig load -d devel` or `JTL_CONFIG_DIR` points at the config directory.
- **Main Flow**:
  1. App's updated `get_config()` locates and reads the `.env` file.
  2. `os.environ` values override any file-loaded values (existing precedence preserved).
  3. `DATABASE_URI`, `DOCKER_URI`, OAuth client credentials, and all tokens resolve to correct values for the target environment.
  4. SSH key resolves: decoded to `/root/.ssh/id_rsa` by the entrypoint in prod; available as a file reference in local devel.
- **Postconditions**: App authenticates users and provisions code-server hosts without missing-config errors across devel, local-prod, and prod.
- **Acceptance Criteria**:
  - [ ] App starts without config errors in devel, local-prod, and prod.
  - [ ] Login flow completes (Google OAuth or password) in each environment.
  - [ ] A code-server host-start succeeds (DOCKER_URI and SSH key both resolve).

## SUC-004: Pre-commit hook prevents future plaintext secret commits

- **Actor**: Developer
- **Trigger**: Developer runs `git commit`.
- **Preconditions**: `dotconfig install-hooks` has been run; the pre-commit hook is active.
- **Main Flow**:
  1. Developer stages changes and runs `git commit`.
  2. Pre-commit hook executes `dotconfig audit`.
  3. If any unencrypted secret values are detected, the commit is blocked with an error.
  4. Developer encrypts or removes the offending content, then re-commits.
- **Postconditions**: No plaintext secrets are committed to the repository.
- **Acceptance Criteria**:
  - [ ] `dotconfig install-hooks` installs the hook in `.git/hooks/pre-commit`.
  - [ ] A commit containing a plaintext secret value is blocked by the hook.
  - [ ] A clean commit (no secrets) passes the hook.
